## SPAM WRITERS / POW PERF — root cause = slow grind, NOT chain-specific (2026-06-07)
User reported "spammers seem down." They are NOT down. First hypothesis ("mainnet PoW is more
expensive") was WRONG and disproved by metrics — recorded here so the mistake isn't repeated.
DATA (one instrumented doPoW+addMessage per chain, run inside the spam containers):
  chain 943: difficulty=173372 iterations=211007 duration=135.5s -> 1557 hashes/s -> addMessage OK
  chain 1:   difficulty=173172 iterations=468526 duration=293.5s -> 1596 hashes/s -> addMessage OK
- msgboard_status on all 3 chains: IDENTICAL workMultiplier=10000 workDivisor=1000000 (difficulty
  ~173k everywhere). User was right: no difficulty difference.
- IDENTICAL grind rate (~1560 hashes/s) on every chain. chain-1 just drew 2.7x the mean iterations
  this run vs 943's 1.2x — normal geometric-distribution variance, NOT a chain-1 bug.
- Block heads are STABLE/monotonic on all chains (sampled 20x/10s) — the rooting block is not thrashing.
ROOT CAUSE: the PoW hot path is NOT sha256-bound. core/src/utils.ts getChallenge() does a FULL
elliptic-curve scalar multiplication `g.mul(nonce)` EVERY iteration (elliptic/bn.js, ~0.6ms each),
capping the grind at ~1560/s. At difficulty ~173k that's ~110s PER MESSAGE on average — but the spam
interval is 60s, so a single grinder can NEVER keep ANY board full. chain 1 just looks emptiest
(lowest volume + unlucky variance). All boards are undersupplied, not just Ethereum.
FIX (in progress, this session): the challenge scalar grows by a CONSTANT `digest` per nonce
(nonce+=1), so the challenge POINT advances by a constant point D = g·digest each step. Replace the
per-iteration scalar MULTIPLY with a running point ADDITION (challenge += D), rebasing once per block
change. Bit-identical challenges (group homomorphism g·a+g·b=g·(a+b), and g·x depends only on x mod n),
so it stays consensus-compatible; ~20-50x faster (->~3-6s/msg). Plus SPAM_WORKERS in spam.ts
(worker_threads) for multi-core parallelism. checkWork/getChallenge stay UNCHANGED as the canonical
verifier — only the SEARCH loop is optimized.
DIAGNOSTIC RECIPE (reusable): docker stats (grinding=~99% CPU vs hung=~0%); msgboard_status per chain
via `docker exec spam-$c node -e "fetch(process.env.RPC_$c, ...msgboard_status...)"` (no curl in the
node:slim image — use node fetch); newest archived msg per chain via the GraphQL max(first_seen_at);
to time a real grind, docker cp a probe .mjs into /app (NOT /tmp — ESM resolves node_modules from the
file's dir) and `docker exec -w /app spam-$c node /app/probe.mjs`.

## content → data_text RENAME — DEPLOYED + VERIFIED (2026-06-07, commit 84b70ed)
Renamed the archive's decoded-message-data column `content` → `data_text`. Why: `content` broke the
`category`/`category_text` pairing and collided with the protocol's own `msgboard_content` meaning
(the board's whole contents). Now the decoded twin always mirrors its raw column and matches the node's
own `data` field key (Message/RPCMessage in @msgboard/core). User directive: match existing/node terms,
don't invent new vocabulary. Final archive convention: category/category_text + data/data_text.
- Code (committed): @msgboard/history archive.ts (CREATE TABLE col, INSERT, SELECT, ArchivedMessage
  type, `contains` filter clause), server.ts + history/relayer READMEs + docs/graphql-archive.md.
  16 history + 52 relayer tests green.
- SELF-HEALING MIGRATION: migrate() runs an information_schema-guarded `ALTER TABLE ... RENAME COLUMN
  content TO data_text` — at most once, no-op on a fresh table. The indexer renames the column AND
  starts writing the new name in the SAME restart, so there is no window where it inserts into a
  missing column. (postgres-idempotent-data-migrations skill is for UPDATE/data migrations w/ a marker
  table — a column RENAME is naturally idempotent via the existence guard; matches the file's existing
  IF NOT EXISTS DDL style.)
- DEPLOY made self-healing for Hasura (site.yml, gitignored): wait-probe now waits for the `data_text`
  COLUMN (proves the rename ran, no race) → reload_metadata (so Hasura sees the renamed column) →
  DROP + recreate the anonymous select permission (a bare create returns 400 and would keep the stale
  `content` column) with `data_text`. Also updated the GraphiQL default query (graphiql/index.html).
- VERIFIED LIVE: GraphQL exposes data_text, no longer content (`field 'content' not found`); an existing
  row kept its decoded text (data preserved, not dropped); GraphiQL default query flipped to data_text.

**"Write for me" push-based relay pattern** — two new exports in `@msgboard/relayer`:

- `packages/relayer/src/sources/http-queue.ts` — `httpQueueSource<T>`: a `RelayerSource` backed
  by a Node.js `http.Server`. Clients POST pre-computed RLP (or any payload); the server buffers
  items in memory and drains them per heartbeat tick. Hardened against four security findings:
  - Queue cap: 1000 items max; returns HTTP 429 when full.

**Foundry invariant tests** (`packages/foundry/test/PoWGate.invariant.t.sol`) — 2 invariants passing,
256 runs × 128 000 calls each (~662 s locally, ~5m44s in CI including build):
- `invariant_used_stamps_never_cleared` — once `gate.used(wh)` is true it must remain true.
- `invariant_min_difficulty_immutable` — `gate.minDifficulty()` never changes.

  Handler design: `enterGolden()` submits the canonical vector; `enterAlt(nonceOffset)` submits
  a message with a shifted nonce. Tracks `goldenHashUsed` (set on any successful entry whose
  `workHash` equals `goldenHash`) so the invariant can assert the stamp is never cleared.

  A THIRD invariant (`invariant_only_golden_stamp_can_be_used`) was removed: the fuzzer found
  that nonce periodicity over the secp256k1 curve (curve order
  `NN = 0xFFFF…BAAEDCE6AF48A03BBFD25E8CD0364141`) means two nonces differing by NN produce
  identical `workHash`. Adding NN to the golden nonce yields a different message that passes
  `verify()` and sets `gate.used(goldenHash) = true` — without the original golden message
  ever having been submitted. The invariant's assumption that only the golden message can consume
  the golden stamp was wrong. The handler documents NN as a named constant and the behavior is
  correct (replay protection still holds: the stamp can only be consumed once).
