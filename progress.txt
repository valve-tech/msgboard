RAFFLE MECHANICS — CONFIRMED WITH USER THIS SESSION:
- Scoring: entropy draws a number in a fixed range [0..256] (configurable; the seed reduced into
  range). Player whose REVEALED guess is closest in LINEAR absolute distance wins the pot. Ties broken
  deterministically. (NOT a fixed 256 target — the draw is the target; entropy is what makes it fair.)
- Entries: a player may submit MANY guesses per round; EACH guess is its own ticket with its own equal
  stake. Intentional economics (user's words): per-guess staking DISINCENTIVIZES spamming the number
  space — extra tickets cost more than the marginal payout edge, so flooding is self-defeating.
- Trigger / cadence: a player CREATES a game instance; the FIRST entrant defines its parameters
  (stake, threshold M, range, period). The period (e.g. hourly) is an EVALUATION CHECKPOINT, NOT a
  refund deadline: if M not met by the boundary, the round ROLLS FORWARD to the next period and keeps
  waiting (funds stay escrowed, nobody kicked) until M have entered. Individual `cancel` for a refund
  is the escape hatch (mirrors coin-flip's unmatched-entrant cancel). Effective trigger = the M-th
  commit.
- LIQUIDITY RULE (load-bearing): we must NUDGE everyone toward a SMALL SET OF CANONICAL PARAMETER
  PRESETS ("easy buttons") so players concentrate into the SAME games. Fragmentation is the enemy — a
  raffle needs M players to fill, so bespoke param tuples strand people. Likely: contract supports
  parameterized games keyed by their param tuple; UI surfaces a few canonical tuples prominently +
  an advanced/custom path. (Confirm enforcement: UI nudge + on-chain recommended-presets list vs hard
  whitelist — OPEN.)

TWO LAYERED COMMIT-REVEALS (key design insight, distinguishes raffle from coin-flip):
  1. VALIDATOR/entropy commit-reveal = the existing ink(commit)->heat(consume)->cast(reveal->seed)
     lifecycle. Produces the draw. Same as coin-flip.
  2. PLAYER commit-reveal = each ticket commits hash(guess, salt) at entry; AFTER the seed is set,
     contenders reveal (guess, salt), contract verifies the hash + computes distance. Ordering: commits
     lock at threshold BEFORE heat; seed set by cast AFTER; reveals after seed (can't change a committed
     guess). Non-reveal = forfeit (lost stake stays in pot) — only likely-winners bother to reveal
     (gas), the closest REVEALER takes the pot. No validator fallback for player reveals (validators
     don't know players' salts), unlike coin-flip's walk-away/validator-cast fallback.

UNIFYING ABSTRACTION (why scope C works): both games = "join the open instance keyed by your
parameters; when a fill condition is met, consume entropy; settle by a scoring rule."
  - coin-flip: match into a PAIR by (side, stake); fill = 2 opposite sides; score = seed parity.
  - raffle:    match into a SHARED ROUND by (stake, threshold, range, period); fill = M commits at a
               checkpoint; score = closest linear distance to the draw.

COLLUSION ANALYSIS (verified against Random.heat, lines 347-428) — THE LOAD-BEARING MECHANISM FACT:
heat selection is CALLER-CONTROLLED. Operator passes potentialLocations; heat walks them in order and
takes the first `required` that _ignite. NO protocol-enforced provider diversity, NO forced inclusion,
NO selection randomization. key = locations.hash() (location POINTERS); seed = hash(revealed secrets)
set at cast. => Whoever controls the selection AND knows the secrets behind it KNOWS THE DRAW at heat
time.
THE ATTACK on a validator-only draw (SELECTION-GRINDING): operator inked the pool so knows all its
secrets; controls selection; draw is only 257 values [0..256]; off-chain BEFORE submitting heat it
searches subsets of its known preimages for one whose hash(secrets) mod 257 ~= a confederate's
committed guess g; submits that one selection; confederate wins every round. Cheap, offline, no
on-chain trial, undetectable. PLAYER COMMIT-REVEAL does NOTHING against it (operator steers draw TO the
confederate's guess, doesn't need others' guesses). UNDERMINE does NOTHING against it (undermine =
reveal-timing past expiry; this happens at heat, pre-reveal).
TWO DEFENSES (must remove selection-control OR secret-knowledge): (1) ENFORCED SELECTION DIVERSITY —
GameBase requires heated set to span N DISTINCT registered validators (check each location's provider);
safe iff >=1 of N honest. Implementable on top of heat (contract validates potentialLocations). Makes
"entropy out of user's reach" safe BUT is an explicit validator-trust assumption -> must be disclosed
in UI. (2) PLAYER-CONTRIBUTED ENTROPY — fold 1 committed undermine-protected player secret into seed;
operator can't know it at selection time -> can't grind -> fair even if ALL validators collude. But
player holds entropy (against the default).
UNESCAPABLE TRADEOFF: "user holds zero entropy" and "user trusts zero validators" are MUTUALLY
EXCLUSIVE. Entropy-out-of-reach default => player necessarily trusts validator set => that default MUST
ship Defense 1 + disclosure. Trustless-vs-all-validators => player must hold a secret (Defense 2).
LATENT: coin-flip both-players-walk-away (both hash(1)) has the IDENTICAL grind hole (seed =
hash(1,1,validatorsecrets)). So ENFORCED SELECTION DIVERSITY belongs in GameBase regardless of game —
it's the safety floor whenever no player contributes entropy.

UNDERMINE-IS-NOT-A-GRIND — DEFINITIVELY VERIFIED (user asked: "would undermining just be ~512 hashes
until the same result is achieved even if we can't find their hashed secret?"). ANSWER: NO for
value-grinding. Grounded in Random.sol _flick (line 132): `if (formerSecret.hash() != _toPreimage(info))
revert SecretMismatch()`. _toPreimage reads the preimage from the INKED SSTORE2 pointer (_pointers[...])
= WRITE-ONCE bytecode = IMMUTABLE from ink time. seed = revealed.hash() over these _flick-verified
values. So injecting a CHOSEN value into a seed = reversing keccak256 = 2^256, NOT ~512. Every actor's
freedom is BINARY (reveal the pinned secret, or withhold) — never "dial in a value". Undermine
(Consumer._undermineExpired) overwrites CONSUMER's own _preimage[id] map, NOT Random's inked SSTORE2
preimages, so it CANNOT inject a chosen value into a draw; its only power is the stall deterrent. =>
Path B (undermine) adds NO hash-grinding hole.
BUT user's general instinct is RIGHT about the surface that IS grindable: SELECTION, not values. 257
possible draws => anyone who knows the secrets AND chooses which immutable preimages enter the seed
grinds the draw in a few hundred OFFLINE tries (the operator selection-grinding attack). Defended by
enforced selection diversity / player entropy. KEY: selection is LOCKED AT HEAT (cast requires
key==locations.hash(), key fixed at heat) — so the defense is "constrain what selection heat accepts,"
a GameBase up-front check; undermine is orthogonal (can't unlock selection post-heat). Net: undermine =
safe to adopt, does what user wanted (kills stall-to-abort, no player-chooseable re-roll), no grind
surface; grinding risk fully carried by the selection-diversity requirement.

*** MAJOR REVERSAL (supersedes the Path B / undermine decision above) — UNDERMINE IS UNSAFE FOR
SMALL-OUTPUT GAMES ***
User's insight (the ~512-hash point, finally understood correctly): undermine is the CONSUMER
REPLACEMENT path — _undermineExpired sets _preimage[id] = keccak(a value the CALLER CHOOSES). Not
reversing keccak (2^256) — FORWARD-hashing chosen replacements (cheap). Timing kills it: undermine is
allowed only AFTER the N validators have RELEASED, so all other seed inputs are known/fixed; the only
free variable is the replaced contribution. Grinder loops v=0,1,2,... computes output=reduce(known
secrets, v), stops at the desired output. Output space makes it trivial: RAFFLE 257 draws -> ~hundreds
of tries ("up to ~512"); COIN-FLIP parity = 1 BIT -> ~2 tries. So undermine doesn't re-roll randomly
for a small-output game — it lets the trigger DICTATE the outcome. STRICTLY WORSE than stall-to-abort
(Path A lets a loser abort; Path B lets a griefer choose the winner). PATH B IS DEAD.
THE GENERAL PROOF (both my earlier questions, combined): ANY player-held contribution to a small-output
seed is exploitable — PINNED (Path A) => last revealer withholds to abort a loss; UNDERMINABLE (Path B)
=> griefer forward-grinds the replacement to fix the result. No third option for a player-held seed
input over a 2- or 257-value output. => The ONLY safe design: NO player (and nothing replaceable) feeds
the small-output seed. This is the "entropy out of the user's reach" instinct, now a SECURITY
REQUIREMENT not a UX nicety.
NOTE: undermine/Consumer remain valid for LARGE-output consumers (grinding a specific 256-bit seed is
infeasible) — just not for our tiny-output games. Phase-1's instinct to SKIP Consumer was correct; the
Path B detour was wrong.

- Core Random/Consumer UNTOUCHED. Game logic lives in a NEW PERIPHERY contract (CoinFlip.sol, the

- TERMINOLOGY (was the confusion): ink = publish/create the contract-full-of-preimages (the commit, like
  invisible ink); heat = consume/select preimages (request, returns key); cast = reveal secrets -> seed.
  Validators INK an always-on pool; a flip HEATs ~3 of them (+ player preimages); winner/validator CASTs.
- TWO POTS, kept separate: (a) Random's own provider economics (stake return via _flick + seed-selected
  provider gets `total` bonus — this is how validators get paid for entropy; VERIFIED by a new test, see
  below); (b) the PLAYER wager pot, ESCROWED in the periphery contract and paid to the winner on settlement.
- ESCROW-ON-ENTRY (not allowance-pull): stake taken when the player joins the queue, so nothing can fall
  through at pairing time. Recovery = cancel/refund an unmatched entry; timeout-refund a paired-but-never-
  finalized flip; walk-aways need no refund (validator fallback cast finalizes).
- MATCHING BY SIDE: per-side FIFO queues keyed by stake; an entrant matches the oldest opposite-side entry
  at equal stake (handles "5 want heads, 0 want tails" — they queue until tails arrive).
- WALK-AWAY: a player who won't manage a secret supplies the well-known hash(0); its secret (zero) is
  public so a validator can reveal/cast it for them -> flip still settles.
- SETTLEMENT via the onCast callback: heat sets settings.provider=address(CoinFlip) + callAtChange=true, so
  Random calls onCast(key, seed) back into the contract -> parity of seed picks winner -> pays pot. No
  separate claim step; winner's incentive is simply to make cast happen.
- v1 defaults: native tPLS only; seed-parity outcome (50/50); fee 0; 3 validator inputs; price-0 player +
  validator preimages (only the escrowed wager moves value).

  chain 1:   difficulty=173172 iterations=468526 duration=293.5s -> 1596 hashes/s -> addMessage OK

ROOT CAUSE: the PoW hot path is NOT sha256-bound. core/src/utils.ts getChallenge() does a FULL
elliptic-curve scalar multiplication `g.mul(nonce)` EVERY iteration (elliptic/bn.js, ~0.6ms each),
capping the grind at ~1560/s. At difficulty ~173k that's ~110s PER MESSAGE on average — but the spam
interval is 60s, so a single grinder can NEVER keep ANY board full. chain 1 just looks emptiest
(lowest volume + unlucky variance). All boards are undersupplied, not just Ethereum.
FIX (in progress, this session): the challenge scalar grows by a CONSTANT `digest` per nonce
(nonce+=1), so the challenge POINT advances by a constant point D = g·digest each step. Replace the
per-iteration scalar MULTIPLY with a running point ADDITION (challenge += D), rebasing once per block
change. Bit-identical challenges (group homomorphism g·a+g·b=g·(a+b), and g·x depends only on x mod n),
so it stays consensus-compatible; ~20-50x faster (->~3-6s/msg). Plus SPAM_WORKERS in spam.ts
(worker_threads) for multi-core parallelism. checkWork/getChallenge stay UNCHANGED as the canonical
verifier — only the SEARCH loop is optimized.

via `docker exec spam-$c node -e "fetch(process.env.RPC_$c, ...msgboard_status...)"` (no curl in the

## SESSION 2026-06-06: "Write for me" relay, version 0.0.30, Foundry fuzz+invariant tests — COMPLETE

### Accomplished

**"Write for me" push-based relay pattern** — two new exports in `@msgboard/relayer`:

- `packages/relayer/src/sources/http-queue.ts` — `httpQueueSource<T>`: a `RelayerSource` backed
  by a Node.js `http.Server`. Clients POST pre-computed RLP (or any payload); the server buffers
  items in memory and drains them per heartbeat tick. Hardened against four security findings:
  - Queue cap: 1000 items max; returns HTTP 429 when full.

  a message with a shifted nonce. Tracks `goldenHashUsed` (set on any successful entry whose

  A THIRD invariant (`invariant_only_golden_stamp_can_be_used`) was removed: the fuzzer found

  the golden stamp was wrong. The handler documents NN as a named constant and the behavior is
  correct (replay protection still holds: the stamp can only be consumed once).
