Cosign — docs

← back to the app

Cosign co-signs a Gnosis Safe transaction off-chain, walletless-for-readers and gasless-until-execution, coordinated over MsgBoard — a proof-of-work-stamped message board where posts are cheap to verify, expensive to spam, and ephemeral (the live window is ~120 blocks). No server ever holds a signature. Every owner signs locally with their own wallet, posts the signed share to the board, and any participant can read the shares, verify them against the Safe's own on-chain rules, and execute once threshold is met.

§1 What cosign is

Cosign replaces the thing most multisig UIs quietly centralize: the place signatures live before a transaction has enough of them to execute. Normally that's a vendor's database (a "transaction service"). Cosign uses MsgBoard instead — the same trustless, PoW-gated board used elsewhere in this repo — as a dumb, ephemeral, permissionless relay. A signature share is just a small ABI-encoded blob posted under a rotating category key; anyone (not just the owners) can read it back, and nothing about its validity depends on who's serving the read.

The three moving pieces:

  • cosign-web — the wizard UI (this app) that drives the whole flow: pick a Safe, build/paste a transaction, simulate it, sign, post, collect, execute.
  • @msgboard/cosign — the framework-agnostic SDK: category-key math, the generic SignatureRecord wire format, aggregate(), and pluggable per-backend CosignAdapters (Safe today; Safe4337Module and a Rhinestone ownable-validator adapter also ship in the package).
  • Two small **read-side** services that make the UI convenient, neither of which is trusted for anything — safe-indexer (owner → Safes discovery on chains with no official Safe Transaction Service) and cosign-archive (keeps serving shares after they've scrolled off the live board's window).

§2 The trust model

Three independent guarantees make the "no trusted server" claim actually hold:

A

The digest is proven byte-equal to the Safe's own hash

The EIP-712 typed-data table the UI signs (SAFE_TX_TYPES in packages/cosign-web/src/lib/safe-typed-data.ts) is a hand-copied mirror of the SDK's internal table — necessary because the SDK doesn't export the raw typed-data shape, only the resulting digest function. To make sure that copy can never silently drift, every signature is checked with assertSafeTxSignatureParity() before it's posted: it recomputes safeTransactionDigest() from the SDK's canonical, independently-implemented hasher (packages/cosign/src/adapters/safe.ts, matching Safe's on-chain SAFE_TX_TYPEHASH/DOMAIN_SEPARATOR_TYPEHASH — no name/version in the domain, exactly like Safe's own domainSeparator()), then recovers the signer from the signature over that digest. If the recovered address doesn't match the wallet that just signed, the app throws and refuses to post. A drifted local types table can never produce a share the Safe adapter would later silently reject.

B

Signatures are verified against the Safe's own membership rules — not trusted

Nothing about a posted share is taken on faith. The CosignAdapter for Safe (makeSafeAdapter) reads getOwners()/getThreshold() straight off-chain and, for every candidate share, recovers the effective signer and checks it's a current owner before counting it. EIP-1271 (contract-owner) shares are checked the same way Safe's own checkNSignatures would: rebuilding the exact pre-image and calling the owner contract's isValidSignature. Verification failures are excluded, not trusted-through.

C

You see the real effect before you sign

Before asking for a signature, the app simulates the exact execTransaction call via eth_simulateV1 on the Safe's own chain, using a state override (not a fake transaction) — temporarily setting threshold = 1 and marking the connected wallet an owner in the Safe's own storage layout, so the simulation exercises the Safe's real code path (checkNSignatures's approved-hash branch) with no signatures collected and no chain state touched. Net asset changes (native + ERC-20/ERC-7528) are decoded from the trace and shown as "what this does," with a hard revert flag. Falls back through debug_traceCall → trace_call → calldata-only decode on chains without eth_simulateV1.

§3 The co-sign flow, step by step

1

Connect + pick the Safe

Connect a wallet; cosign-web auto-discovers the Safes that wallet owns (or accepts a manual Safe address). Discovery is a convenience only — it never gates anything. Once a Safe is chosen, its owners/threshold are read straight off the Safe contract, and the coordination scope is fixed: scope = safe:<chainId>:<safe address, lowercased> (scopeFor() in lib/cosign.ts) — this is what buckets shares per-Safe-per-chain so unrelated Safes never collide on the board.

why it's trustless: discovery only shortens a text box; nothing downstream trusts it. Owners/threshold come from the Safe contract itself.
2

Build the SafeTx + compute the digest

Either build a SafeTx (to/value/data/operation/gas params/nonce) or paste a raw 32-byte digest directly. For a built SafeTx, safeTransactionDigest(tx, chainId, safe) computes the EIP-712 hash locally — this is the exact value Safe's on-chain getTransactionHash(...) returns for the same inputs.

why it's trustless: the digest is a pure function of the transaction + chain + Safe address — no oracle, no server round-trip required to know what you're about to sign.
3

Simulate — "what this does"

Runs the state-override eth_simulateV1 described above and shows a plain-language summary, per-address net asset deltas, and a raw-trace expander. A reverting transaction is flagged and blocks the "confirm & continue" button.

why it's trustless: you're looking at the Safe's real bytecode executing against real state overrides, not a vendor's description of the calldata.
4

Sign — with the parity guardrail

The wallet signs the EIP-712 typed data (or, for a pasted digest, an eth_sign-style ECDSA signature — recorded with scheme = ECDSA rather than EIP712). For a built SafeTx, assertSafeTxSignatureParity() runs immediately after signing and before anything is posted (§2.A above).

why it's trustless: the guardrail makes "the app signed the wrong hash" a build-time-impossible failure mode rather than a silent one.
5

Post the share to MsgBoard

The signature is packed into a SignatureRecord — { digest, signer, signature, scheme, meta } (for a SafeTx, meta ABI-encodes the full tx so any reader can later reconstruct execTransaction's arguments) — ABI-encoded per RECORD_ABI, and posted under today's rotating category key. The proof-of-work stamping this requires runs in a Web Worker — never the browser's main thread (packages/cosign-web/src/seams/worker-board.ts) — so the UI stays responsive while the share is minted onto the board.

why it's trustless: posting is a public write to a PoW-gated board, not a write to anyone's private database — anyone can independently confirm the post happened and reproduce its decode.
6

Read, union, and aggregate

Any participant reads the rolling 7-UTC-day window of category keys (WINDOW_DAYS in lib/cosign.ts) for the scope, unioned across the live board and the cosign-archive service — board records win on a signature collision, and the archive only contributes shares that have aged out of the board's live window (dedupe key: digest:signature, lowercased). The union degrades gracefully to board-only if the archive is unreachable. Once shares for the chosen digest are in hand, the SDK's aggregate() filters them through the Safe adapter's verify() (recover signer → confirm current owner membership, or the EIP-1271 pre-image check), keeps only the ones that pass, and orders the survivors the way Safe's checkNSignatures requires (strictly ascending by signer address). aggregateForSafe() additionally builds the final signatures bytes blob ready for execTransaction.

why it's trustless: a share only counts if it independently verifies against the Safe's live owner set — a malicious or buggy read source can at worst withhold a valid share, never inject a fake one that passes verification.
7

Execute

Once signed owners ≥ threshold, any owner (or anyone holding the assembled blob) submits execTransaction(to, value, data, operation, safeTxGas, baseGas, gasPrice, gasToken, refundReceiver, signatures) directly to the Safe and pays the gas. Cosign itself never touches the chain on the write side before this point — every prior step is board coordination only.

why it's trustless: execution is a plain call to the Safe contract, which re-verifies every signature itself (checkNSignatures) before moving any funds — cosign's own verification is a courtesy for the UI, not the actual security boundary.

§4 Create a Safe (v1.4.1)

Cosign can also deploy a brand-new Safe rather than requiring one to already exist. This uses the canonical, deterministic Safe v1.4.1 deployment (same addresses on every EVM chain):

roleaddress
ProxyFactory0x4e1DCf7AD4e460CfD30791CCC4F9c8a4f820ec67
Safe singleton (L2)0x29fcB43b46531BcA003ddC8FCB67FFE91900C762
Fallback handler0xfd0732Dc9E303f09fCEf3a7388Ad10A83459Ec99
1

Predict

Given owners + threshold + a random 256-bit saltNonce, predictSafeAddress() computes the deterministic CREATE2 address purely and synchronously — no RPC call — from the setup() initializer calldata, the embedded canonical SafeProxyFactory creation code, and the factory address. The predicted address is shown live as the owners/threshold form is edited.

2

Deploy

The connected wallet calls createProxyWithNonce(singleton, initializer, saltNonce) on the factory. This mines the proxy and emits ProxyCreation(proxy, singleton).

3

Verify (mined == predicted)

confirmDeploy() waits for the receipt, decodes the ProxyCreation log, and throws (SafeAddressMismatchError) if the address that actually got deployed doesn't match the address predicted in step 1 — the app never hands a Safe address downstream that it hasn't independently confirmed came from the exact deployment it asked for.

4

Hand off

The confirmed Safe address (pinned to the chain the deploy actually ran on, not just "whatever chain the wallet happens to be on by the time this resolves") is handed straight into the co-sign flow at step 1 — owners/threshold are then re-read from the freshly deployed contract, same as any manually-entered Safe.

§5 Gasless deploy (optional)

Step 2 of Create-a-Safe above (createProxyWithNonce) normally costs the connected wallet gas. cosign-relay is a small, purely-additive service (packages/cosign-relay) that will submit that same call and pay the gas itself, for a caller with an empty wallet — nothing else about the flow changes: prediction (step 1) and the mined-address verification (step 3) are identical either way, and CREATE2 addresses are submitter-independent, so a relay paying the gas can never change what address a given (factory, singleton, initializer, saltNonce) deploys to. The toggle only appears when the connected chain is one the relay currently sponsors.

1

Sign a request digest — not a SafeTx

The browser computes the same predicted address as step 1, then builds a request digest that binds the exact deploy request: keccak256(abi.encode(chainId, singleton, keccak256(initializer), saltNonce)). The connected wallet signs this raw 32-byte digest with a plain EIP-191 personal-sign (not the EIP-712 SafeTx signing used elsewhere in cosign) — binding chain id + singleton + initializer + saltNonce means the signature can never be replayed against a different chain, singleton, or Safe configuration.

2

Solve a small proof of work

A walletless hashcash-style stamp over the same digest: find a 32-byte nonce such that keccak256(digest ++ pad(nonce, 32)), read as a uint256, falls below a difficulty target (powBits, from the relay's own /config; default 20 — roughly a second of grinding). This runs in a Web Worker, never the browser's main thread, matching the same off-thread-PoW convention used for posting shares to MsgBoard.

3

POST to the relay

{ chainId, singleton, initializer, saltNonce, signature, powNonce } goes to POST /relay/deploy-safe. The relay checks, cheapest-first:

  • the chain is one it currently sponsors, and singleton is the canonical Safe v1.4.1 L2 singleton;
  • the setup is a plain owners+threshold multisig only — to == 0x0, data == 0x, the canonical fallback handler, paymentToken/ payment/paymentReceiver all zero. Any setup that would delegatecall out or redirect a payment is rejected outright — this is what makes it safe for the relay to pay gas at all, since it can never be tricked into sponsoring anything but a Safe's own ownership set;
  • Gate 1 — proof of work (step 2 above) is verified;
  • Gate 2 — signed owner: the signature (step 1 above) must recover to one of the setup's own owners[] — a stranger can never spend the relay's gas;
  • Gate 3 — rate limit: a per-IP daily cap (default 5/day) as a coarser backstop.

Only after all of that does it submit createProxyWithNonce from its own per-chain funded key and return { txHash, proxy }.

4

Verify — exactly like the user-pays path

The app runs the same confirmDeploy() against the relay's txHash as step 3 of Create-a-Safe, throwing on any mismatch between the mined and predicted address. The relay's own validation protects its gas, not this guarantee — the app never trusts the relay for anything beyond "a transaction got submitted."

§6 Architecture

cosign-web (this app)
React wizard: pick Safe → build/simulate tx → sign → post → collect → execute. No backend of its own.
@msgboard/cosign SDK
Category-key math, SignatureRecord codec, generic aggregate(), CosignAdapter (Safe / Safe4337 / Rhinestone).
MsgBoard
PoW-gated message board. Posts are the ONLY write path for coordination — ephemeral, ~120-block live window.
safe-indexer
Ponder-indexed owner→Safe map for chains with no official Safe Transaction Service (PulseChain 369/943). Read-only convenience.
cosign-archive
Serves decoded shares that scrolled off the board's live window, keyed identically to the board (namespace/scope/day). Read-only convenience.
the Safe contract
The actual security boundary — re-verifies every signature in checkNSignatures at execTransaction time regardless of what the UI already checked.

The MsgBoard the shares travel over is independent of the chain the Safe lives on — cosign is a pure coordination layer with no chain writes of its own, so the board can run on any EVM chain exposing the msgboard_ RPC module (packages/cosign-web/src/lib/board.ts).

The category-key scheme

Every share lives under a rotating, day-granular UTC category key so the board's working set stays small and self-pruning: 

categoryKey = keccak256(utf8Bytes(`${namespace}:${scope}:${isoDate}`))
namespace   = "cosign"
scope       = `safe:${chainId}:${safeAddress.toLowerCase()}`
isoDate     = YYYY-MM-DD, UTC (e.g. "2026-07-02")

Readers sweep a rolling window (cosign-web uses 7 days) of these keys — today's plus the prior N−1 UTC days — across both the live board and the archive, decode every entry, skip anything undecodable (the board is open; junk under a category is expected and simply ignored), and dedupe by keccak256 of the raw ABI bytes (board reads) or by digest:signature (board∪archive union).
